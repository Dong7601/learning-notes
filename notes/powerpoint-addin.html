PowerPointアドイン開発記 — 保存失敗の真因を切り分ける

既存業務アドイン風PowerPointアドインの開発で、保存失敗、信頼設定、セキュリティ製品隔離が連鎖した。結論は「OOXML保存の破損」ではなく、1箇所の型宣言と環境ポリシーの分離だった。

  • VBA + RibbonX
  • PowerPoint .ppam
  • E_FAIL
  • 遅延コンパイル
  • セキュリティ製品

TL;DR

まず結論。長いデバッグ叙事詩の中心は、壊れた保存経路ではなく、壊れた前提だった。

  1. 目標は、リボン「PowerPoint部品アドイン」からテンプレスライドと図形グループ要素をサムネイルギャラリーで挿入する既存業務アドイン風PowerPointアドインだった。
  2. 保存が延々とE_FAILで失敗し、VBOM設定、StrPtr、サイズ、モジュール数、危険構文、保存経路を切り分けた。
  3. 真犯人は Dim w As Window。PowerPointに存在しないExcel由来の型を1箇所だけ使っていた。
  4. As DocumentWindow に修正後、COMでフル実VBAの .ppam は生成できた。
  5. 最後はセキュリティ製品が実コードの .ppam を隔離し、管理PCでは動かせなかった。
保存失敗の真犯人 1単語の型宣言 Window → DocumentWindow

そもそも何を作っていたのか

VBAやマクロを知らない前提で、PowerPoint部品アドインの正体を先に整理する。

PowerPointアドイン

PowerPointに追加機能を差し込む小さな拡張パッケージ。今回のPowerPoint部品アドインは、PowerPointのリボンに「PowerPoint部品アドイン」タブを出し、用意済みのスライド部品をワンクリックで挿入する道具だった。

配布形式は .ppam。PowerPointに読み込ませると、通常のプレゼンとは別に裏側で常駐し、ボタンやギャラリーを提供する。

マクロ

Office上の操作を自動化する命令のまとまり。人間が手で行う「スライドを開く」「図形をコピーする」「現在のページへ貼る」といった手順を、ボタン一発で実行させる。

セキュリティ上は強い権限を持つため、管理PCでは信頼場所、署名、セキュリティ製品、マクロポリシーの影響を受ける。

VBA

Visual Basic for Applicationsの略。Excel、PowerPoint、Wordなどに組み込まれているマクロ用のプログラミング言語。

注意点は、Officeアプリごとに使える型やオブジェクト名が違うこと。今回の Window はExcelでは自然でも、PowerPoint VBAでは正しい型名ではなかった。

RibbonX

OfficeのリボンUIを定義するXML。どのタブに、どのボタンやギャラリーを置き、クリック時にどのVBA関数を呼ぶかをつなぐ。

見た目の入口がRibbonX、実際に動く処理がVBA、配布の箱が .ppam という分担だった。

利用者の操作 PowerPointのリボンでPowerPoint部品アドインタブを開き、ギャラリーから部品を選ぶ。
RibbonX リボン上のボタンやギャラリーを定義し、クリック時に呼ぶVBA関数名を指定する。
VBAマクロ ライブラリ用PPTXからスライドや図形を取り出し、現在のプレゼンへ挿入する。
.ppam RibbonXとVBAをPowerPointアドインとしてまとめ、PowerPointに読み込ませる配布ファイル。
COM/VBOM 外部スクリプトからPowerPointやVBAプロジェクトを操作して、ビルドや検証を自動化する経路。

社内配布アドインはどう入っているのか

手元のPowerPointに表示されている社内配布アドインのタブを、ファイル構造から逆算する。

実体は .ppam ファイル

調査した管理PCでは、社内配布アドインの本体がユーザー別Officeアドインフォルダ内の .ppam として配置されていた。標準のOfficeアドイン置き場にあるため、PowerPointが読み込める。

同じ周辺フォルダにReadMeやテンプレートらしきPPTXもあり、ヘルプやテンプレートをアドインから参照する構成に見える。

社内配布アドインのタブはRibbonXで出ている

.ppam の中には customUI/customUI14.xml があり、そこで独自タブ、各グループ、ボタン、メニューが定義されていた。

PowerPoint上に見える独自タブのグループ構成は、このXML側のグループ定義と対応する。

処理本体はVBA

同じ .ppam 内に ppt/vbaProject.bin が入っていた。これはVBAプロジェクト本体で、RibbonXの onAction から呼ばれるマクロ群が入る場所。

たとえばRibbonX上のボタンは onAction="Save_SelectedSlides" のように、クリック時に呼ぶVBA側の関数名を持つ。

PowerPoint部品アドインとの違い

仕組み自体はPowerPoint部品アドインと近い。違いは、社内配布アドインは管理されたPC環境で許可済みまたは既に配布済みの .ppam として置かれている点。

PowerPoint部品アドインは自作の新しい .ppam なので、同じ技術構造でもセキュリティ製品やマクロポリシーから見ると別物として扱われ得る。

配置 ユーザー別Officeアドインフォルダに .ppam 形式のアドイン本体が置かれる。
読み込み PowerPointがアドインとして .ppam を読み込む。読み込まれると、通常のタブとは別に社内配布アドインのタブが増える。
表示 customUI/customUI14.xml がリボンのタブ、グループ、ボタン、アイコンを定義する。
実行 ボタンの onAction が vbaProject.bin 内のVBAマクロを呼ぶ。
補助ファイル 隣接フォルダのReadMeやテンプレートPPTXを、ヘルプやテンプレート機能として参照している可能性が高い。

インタラクティブ・タイムライン

各ステップを開くと、症状、調べたこと、分かったことを確認できる。

症状
リボン「PowerPoint部品アドイン」からテンプレスライドと図形グループ要素を挿入したい。
調べたこと
Office.jsとVBA+RibbonXの実装条件を比較した。
分かったこと
Office.jsは常駐サーバーが必要で、図形グループ挿入も不可だったため、VBA+RibbonXを採用した。
症状
レジストリの AccessVBOM=1 でもVBAプロジェクト オブジェクト モデルへのアクセスが拒否された。
調べたこと
PowerPointとExcelで同じ信頼設定を比較した。
分かったこと
Trust CenterのUIチェックだけが有効で、設定はクラウド同期ストアに保存され、レジストリには現れなかった。
症状
StrPtr を含むVBAモジュールがあると .ppam 保存がE_FAILになった。
調べたこと
GetPrivateProfileStringW と StrPtr を使うINI読み取りを切り分けた。
分かったこと
Win32 INI APIを廃止し、FileSystemObject製の純VBA INIパーサに置換して回避した。
症状
フル6モジュールで .pptm / .ppam 保存がE_FAILになった。
調べたこと
単体モジュール、同程度の大きさのダミー、各種型を使う単体、1モジュール統合、COM SaveAs、SaveCopyAs、VBA内部SaveAs、可視/不可視ウィンドウを試した。
分かったこと
サイズでもモジュール数でも保存経路でもなかった。バイナリ .ppt だけは保存できたが、VBA抽出不能だった。
症状
一度保存に失敗すると、そのPowerPointインスタンスでは以後の保存も全部失敗した。
調べたこと
同じプロセス内で複数ケースを回した結果と、プロセスを分けた結果を比較した。
分かったこと
正しい観測単位は「1プロセス1テスト」だった。
症状
保存はE_FAILだが、VBA自体の一部実行は成功していた。バージョン確認用の小さな関数は正しい値を返した。
調べたこと
壊れた関数まで含めたプロジェクト全体のコンパイル状態を見直した。
分かったこと
Window はExcelの型で、PowerPointの正しい型は DocumentWindow。OOXML保存はコンパイル状態をシリアライズするため、コンパイル不能だと保存がE_FAILになった。
症状
修正前は30回以上の保存失敗テストが全て同じバグ入りコードで行われていた。
調べたこと
型宣言を修正した状態でCOMからフル実VBAの .ppam 生成を確認した。
分かったこと
当初の「OOXML保存が壊れている」という結論は誤り。正しくは「コンパイル不能なプロジェクトは保存できない」。
症状
レジストリの AddIns キーに Path と AutoLoad=1 を設定しても、新規アドインの起動時自動ロードが有効にならなかった。
調べたこと
既存の業務アドインを別名登録して、同じ仕組みでも読み込み条件が変わるかを試した。
分かったこと
PowerPointのアドインUIダイアログから追加したときに記録される「同意」がゲートで、これもレジストリ外ストアに保存された。
症状
管理PCのセキュリティ製品が実コードの .ppam を永続隔離し、ファイルがアクセス拒否でロックされ、PowerPointも読み込めなかった。
調べたこと
サイズ、危険構文、コメント削減、自己署名コード署名、信頼場所でないマクロ実行ポリシーを検証した。
分かったこと
これはコードの問題ではなく管理PCのセキュリティポリシーの壁。現実解はIT部門への許可申請、または制約を承知したOffice.js版への作り直しだった。

犯人はどれだ?

容疑者ボードは、疑い、検証、評決を分けて読むためのもの。カードを開くと詳細が出る。

疑い: AccessVBOM=1 が効かず、VBA操作が拒否される。

検証: Trust Center UIチェックとレジストリ値を比較し、Excelでも確認。

評決: この環境ではレジストリ外ストアが効く環境要因。

疑い: StrPtr を含むVBAモジュールが保存を壊す。

検証: Win32 INI APIを廃止し、FileSystemObject製の純VBA INIパーサへ置換。

評決: 回避対象ではあったが、後続のフル結合保存失敗の真犯人ではない。

疑い: コード量やファイルサイズが保存失敗や隔離の原因ではないか。

検証: 同程度の大きさのダミーコードは保存も通過もできた一方、実コードは隔離された。

評決: サイズではなかった。

疑い: フル6モジュール構成そのものが保存を壊す。

検証: 各モジュール単体はOK、フル結合は1モジュールに統合してもNG。

評決: モジュール数ではなかった。

疑い: Win32 API宣言、FileSystemObject、Dictionary、ネイティブファイルI/Oが隔離の原因ではないか。

検証: これら全部入りでも小さければ素通りした。

評決: 個別の危険構文は無罪。

疑い: 署名がないからセキュリティ製品が隔離している。

検証: 自己署名コード署名証明書を作り、ローカル環境で信頼済みとして扱われるようにして署名した。

評決: それでも隔離。セキュリティ製品は署名ベースではなかった。

疑い: どこかの関数だけがプロジェクト全体のコンパイルを壊している。

検証: Dim w As Window を Dim w As DocumentWindow に修正。

評決: 修正後、COMでフル実VBAの .ppam が問題なく生成できた。

疑い: 実コードの .ppam が管理PCのセキュリティポリシーに止められている。

検証: サイズ、危険構文、コメント削減、デジタル署名を検証したが、隔離は変わらなかった。

評決: コードでは超えられない環境の壁。IT部門への許可申請、またはOffice.js版への作り直しが現実解。

コード before / after

保存失敗の中心は、Excelの型名をPowerPoint VBAで使っていた1行だった。

Before

Dim w As Window

After

Dim w As DocumentWindow
1. 存在しない型 Window はExcelの型で、PowerPointには存在しない。
2. プロジェクト不全 その宣言を含むVBAプロジェクトはコンパイル不能になる。
3. 部分的成功の罠 遅延コンパイルにより、呼んだ小さな関数だけは個別にコンパイルされ動いた。
4. 保存E_FAIL OOXML保存はVBAのコンパイル状態をシリアライズするため、保存ごと一般エラーになった。

教訓カード

このケースから取り出せるメタ学習。コードより先に、観測の前提を疑う。

交絡変数を疑え

同じ前提、つまりWindowバグ入りコードで30回テストして得た結論は丸ごと誤りだった。「結論」より「前提」を疑う。

遅延コンパイルの罠

1つの関数が動くことは、プロジェクト全体が健全であることを意味しない。部分的成功を全体の健全性と取り違えない。

1プロセス1テスト

状態が汚染されるシステムでは、毎回まっさらな環境で1要因ずつ変える。

探索法は観測次第

二分探索やleave-one-outは強力だが、汚染下では無効化される。観測装置、つまりテスト環境自体の健全性をまず疑う。

環境の壁を分ける

最後のセキュリティ製品はコードでは超えられない種類の障害だった。コードの壁と環境の壁を分けて考える。

最小再現を作れ

「Aを足すと壊れる」を1要因まで削る。原因が1単語なら、そこまで小さくできる。

用語集

このケースで出てきた言葉を、文脈に必要な最小限で整理する。

ppam

PowerPointアドインとして配布・読み込みするためのファイル形式。

マクロ

Office上の操作を自動化する命令のまとまり。ボタン操作の裏で実行される処理。

VBA

Officeに組み込まれたマクロ用言語。ExcelとPowerPointでは使える型名が一部異なる。

RibbonX

OfficeのリボンUIをカスタマイズするために使った仕組み。

customUI

.ppam 内でリボンのタブ、グループ、ボタン、画像を定義するXML領域。

onAction

RibbonXのボタンがクリックされた時に、どのVBAマクロを呼ぶかを指定する属性。

VBOM

VBAプロジェクト オブジェクト モデル。アクセスには信頼設定が関係した。

OOXML

.pptm や .ppam 保存で関係したOffice文書形式。

COM自動化

PowerPointを外部から操作し、SaveAsなどを実行するために使った自動化経路。

セキュリティ製品

管理PC上で実コードの .ppam を隔離した管理PCのセキュリティ製品。

遅延コンパイル

呼ばれた関数だけが個別にコンパイルされるため、別の壊れた関数を見逃し得る挙動。
