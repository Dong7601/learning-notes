Enterprise AI guide

企業のAI導入：API従量課金 と サブスクの違い — なぜ「全員サブスク」で済まないのか

「全社員にサブスクを配ればいい」という感覚は、人が画面に向かって使う用途では正しい。だが企業がAPIで1日数十万〜数百万円を使うのは、APIが人手を介さず自動でシステムに組み込むためのもので、用途も課金の仕組みも別物だから。さらに「データを海外AI企業に直接渡したくない」という別の理由から、AWSやAzure経由（Amazon Bedrock / Azure OpenAI）で使う構成が選ばれる。この2つの論点を分けて理解する。

まず「2つの軸」を分ける

ユーザーの疑問は、実は独立した2つの軸が混ざっている。ここを分けると一気に整理できる。

軸A：課金モデル

「サブスク定額」か「API従量課金」か。これはコストの話。

軸B：データ経路／インフラ

「AI企業に直接」か「AWS/Azure経由」か。これはセキュリティの話。

両者は独立

この2軸は独立している。巨額コストは軸Aの問題、個人情報の懸念は軸Bの問題で、原因も対策も別。

企業の典型構成

企業の典型構成は「API従量課金（軸A）× AWS/Azure経由（軸B）」。

サブスク と API従量課金 は用途が違う

定額で人が使うものと、プログラムから自動で呼び出すものは、同じAIでも用途が違う。

サブスク

ChatGPT Plus/Team/Enterprise、Claude Pro/Team など。1人あたり定額で、人がチャット画面に向かって対話することが前提。

レート制限（一定時間あたりの回数上限）があり、使いすぎても定額の範囲で頭打ち。

サブスクでは組み込めない

社内システムやアプリには組み込めない。プログラムから叩く用途は規約上APIの領域。

API（従量課金）

処理したトークン量（おおまかには文字量）に応じて課金。上限が無く青天井になりうる。

人手を介さずプログラムから大量に呼べる。アプリ・自動化・社内システムへの組み込み用。

人数と無関係に増える

1人が使うのではなく「1つの仕組み」が動き続ける。人数と無関係にコストが増える。

なぜ1日数十万〜数百万になるのか
  • RAG（社内文書を毎回大量に読み込ませる）で1回のリクエストが巨大化。
  • AIエージェントの自動ループが想定以上に回る／暴走する。
  • バッチ処理・全データ一括処理を人手を介さず実行。
  • リトライや多段推論で見えないところでトークンが積み上がる。

「人が使いすぎた」のではなく「自動化が裏で大量に消費」しているのが典型。設計・監視ミスでも起きる。

なぜ全社員サブスク配布で済まないのか

「全社員に配ればいいのでは」という感覚への直接回答。

日常的な対話利用なら正しい

社員の日常的な対話利用なら、確かに全員サブスク配布が正解に近い。むしろ推奨。

企業が本当にやりたいこと

企業がAIで本当にやりたいことの多くは「既存業務システムへの組み込み・自動化」。これはサブスクでは技術的にも規約的にも実現できず、APIが必須。

現実は併用

現実は「社員の日常利用＝サブスク」＋「プロダクト/自動化＝API従量」の併用になる。

巨額請求はAPI側

巨額請求が話題になるのは後者（API）側。前者（サブスク）はコストが読みやすく事故が起きにくい。

結論：あなたの感覚は「人の利用」については正しい。巨額化するのは別レイヤー（自動化・組み込み）の話。

チャットツール／エージェント導入ならAPIは不要か？

既存システムへの組み込みではなく、AIをチャットツールとして配る・Claude Code / Codex のようなエージェントを導入するだけなら、APIではなくサブスクで足りることが多い。

チャットツールとして導入 → サブスクでOK

社員が ChatGPT / Claude をチャットで使うだけなら、per-seat サブスク（ChatGPT Team/Enterprise、Claude Team/Enterprise）が適切。APIは不要。最初の感覚どおり。

「エージェント導入」＝「API」ではない

Claude Code も Codex もサブスクプランで使える。Claude Code は Claude Pro/Max 等の定額プランに含まれ、Codex も ChatGPT 有料プランに含まれる。人がそのエージェントを手で操作するかぎりサブスク席で導入できる。

APIキーやBedrock/Vertex経由の従量で使う選択肢もあるが必須ではない。

判断の分かれ目は「人が操作するか／無人で動かすか」

人が手で操作（チャットもエージェントも）→サブスクで足りることが多くコストも定額で読みやすい。プログラムから自動・無人で回す／自社システムに組み込む／自社プロダクトに載せる→API。

元ノートの「対話 vs 組み込み」をより正確に言い換えた線引き。

注意：サブスクにも上限はある

エージェントは裏で大量トークンを消費（ファイル読込・多段ループ）。サブスクにも利用上限（レート/週次キャップ）があり、ヘビーに使うと上限に当たる。その場合に初めてAPI従量を検討。「1日数百万」の事故は基本、無人の自動化・API側で起きるもので、人が操作する範囲では起きにくい。

まとめ

あなたの感覚は正しい。チャットツール導入も、人が使うエージェント導入も、サブスクで始められる。 APIが要るのは「無人の自動化」「自社システムへの組み込み」「自社プロダクト化」のとき。

補足として、エージェントでも「データを社外に出したくない」要件があれば Claude Code を Bedrock 経由で使う等の構成になり、その場合はそのクラウドの従量課金になる。課金（軸A）とデータ経路（軸B）が独立なのはエージェントでも同じ。

セキュリティと個人情報

AWS/Azure上に自社サーバーを立てるのは、海外AI企業のサーバーに直接データを入れさせないため、という仮説への回答。

4-1. 仮説の答え

方向性は合っている

方向性は合っている。ただし表現を正確にすると少し違う。

正確にはマネージドAIサービス

実態は「自社サーバーを立てる」というより、クラウド事業者が自社インフラ内でホストするマネージドAIサービスを使う。

AWS の Amazon Bedrock（Claude などをAWS内でホスト）、Azure の Azure OpenAI Service（OpenAIモデルをAzure内でホスト）。

契約・監査済みクラウドの境界

AIモデルを、すでに契約・監査済みのクラウド（AWS/Azure）の境界の中で動かす。

だからデータが Anthropic / OpenAI という別会社のサーバーへ出ていかない。

仮説どおりの狙い

「海外AI企業に直接渡さない」という狙い自体は仮説どおり正しい。

正確には「自社サーバー構築」ではなく「契約済みクラウド内のAIサービス利用」。

4-2. 具体的な仕組み

Amazon Bedrock AnthropicのモデルをAWSが自社データセンター内でホスト。利用者はAWSの自分のアカウント・指定リージョン内でAPIを呼ぶ。入出力データはAWS境界内に留まり、Anthropic側には送られず、モデル学習にも使われない。Anthropicはそのトラフィックを見られない。
Azure OpenAI Service 同様にMicrosoftがOpenAIモデルをAzure内でホスト。データは自分のAzureテナント・リージョン内で処理され、OpenAIには渡らず学習にも使われない。
共通の統制 リージョン指定（データの保管場所を固定）、通信時・保存時の暗号化、IAMによるアクセス制御、VPC/プライベート接続、監査ログ。

データの流れ

AI企業に直接送るルートと、AWS/Azure経由のルートでは、処理されるインフラの境界が変わる。

① AI企業に直接

OpenAI/Anthropic API 直叩き。流れは、自社 → インターネット → OpenAI / Anthropic のサーバーで処理。

現在は両社ともAPIの入力をデフォルトで学習に使わず、ゼロデータ保持の選択肢も提供。ただし「処理は彼らのインフラで行われる」点は変わらない。

② AWS/Azure経由

Bedrock / Azure OpenAI。流れは、自社 → AWS/Azureの自分のテナント・指定リージョン内でモデル推論まで完結。

Anthropic / OpenAI のサーバーには飛ばない。データの宛先が「すでに契約しているクラウド1社」に閉じる。

信頼性の違い

「AI企業に渡すのはNG、AWS/AzureならOK」とされる理由は、AI企業が信用できないという話ではない。

6-1. なぜAWS/AzureならOKとされるのか

大前提（誤解の訂正）：「AnthropicやOpenAIが信用できない」という話ではない。企業のガバナンス・契約・調達の都合という、組織側の合理性。

既存の契約・信頼関係 多くの企業はすでにAWS/Azureと包括契約（データ処理契約・守秘・SLA）を結び監査済み。新たにAI専業ベンダーを審査・契約するコストが要らない。
コンプライアンス認証 AWS/Azureは長年の実績で各種認証（ISO27001、SOC1/2/3、PCI DSS、医療のHIPAA、日本政府調達のISMAP 等）を網羅。法務・調達が承認しやすい。
データレジデンシー リージョンを指定し「データを特定地域（例：日本国内）から出さない」を契約・技術で担保。海外法（米CLOUD法など）への懸念に対応。
既存セキュリティ基盤との統合 IAM・VPC・鍵管理（KMS）・監査ログなど、既に使っている統制の中にAIを組み込める。管理する経路を増やさない。
責任分界・サポート窓口の一本化 問題時の責任範囲と連絡先が既存ベンダーに集約。

6-2. 「AWS/Azureも結局は外部企業では？」への明確な回答

その指摘は正しい。 AWSもAzureも、Amazon/Microsoftという外部企業のサーバーであることに変わりはない。

だから安全性の根拠は「自社所有か外部か」ではない。本質は「データの流れを契約・認証・技術で管理（コントロール）できるか」。

宛先が1社に閉じる

既に契約・監査済みのクラウド内で完結し、AI専業ベンダーという「2社目」を経路に増やさない。管理対象が減る。

契約で縛れる

AWS/Azureとは法的拘束力のあるデータ処理契約があり、「学習に使わない・第三者に渡さない・指定リージョンに留める」を契約で強制できる。

第三者監査で検証できる

上記が認証・監査レポートで客観的に確認できる。

安全性の正体

要するに「外部か内部か」ではなく「コントロールできるか・検証できるか」が安全性の正体。

AI専業ベンダー（Anthropic/OpenAI）もエンタープライズ契約を結べば同等の保証（学習不使用・ゼロ保持・各種認証）を得られる場合がある。それでも企業がAWS/Azure経由を好むのは「すでにある信頼基盤に寄せるほうが、調達・法務・統制が圧倒的に楽」という実務上の理由が大きい。

よくある誤解の訂正

安全性・モデル提供元・課金の論点を混ぜない。

「AI企業＝危険／AWS・Azure＝安全」ではない

単純化しすぎ。両AI企業もエンタープライズ向けに高水準のセキュリティ・認証・ゼロ保持を提供している。選択は「危険回避」より「既存統制への乗せやすさ」。

モデル提供元とインフラ運営者は別

モデルの提供元 と データを処理・保持するインフラの運営者は別概念。Bedrock上のClaudeも中身はAnthropic製、Azure上のGPTも中身はOpenAI製。「誰が作ったモデルか」と「どこで処理・保管されるか」を分けて考える。

安全性とコストは別々

課金（軸A）とデータ経路（軸B）は独立。AWS/Azure経由にしてもAPIは従量課金のままで、巨額化リスクは消えない。安全性とコストは別々に対策する。

用語集

このノートで出てきた言葉を、最小限の文脈で整理する。

従量課金（API）

使ったトークン量に応じて支払う方式。自動化・組み込み向け。上限なく増えうる。

サブスクリプション

1人あたり定額。人が対話で使う前提。レート制限あり。

トークン

AIが処理する文字のかたまりの単位。課金や上限の基準になる。

API

プログラムから機能を呼び出す窓口。人手を介さず大量に呼べる。

Amazon Bedrock

AWSが各社のAIモデルを自社インフラ内で提供するマネージドサービス。

Azure OpenAI Service

MicrosoftがOpenAIモデルをAzure内で提供するサービス。

データレジデンシー

データを保管する地理的場所（リージョン）を指定・固定する考え方。

リージョン

クラウドのデータセンターの地理的区分。日本リージョン等を選べる。

ゼロデータ保持

入力データをベンダー側に残さない設定／契約オプション。

データ処理契約（DPA）

データの取り扱い（学習不使用・第三者非提供等）を法的に縛る契約。

コンプライアンス認証

ISO27001 / SOC2 / ISMAP 等、第三者がセキュリティ水準を保証する仕組み。

RAG

社内文書などを検索して都度AIに読み込ませる方式。トークン消費が大きくなりやすい。

AIエージェント

AIが自律的に複数手順を実行する仕組み。ループでトークンを大量消費しうる。
