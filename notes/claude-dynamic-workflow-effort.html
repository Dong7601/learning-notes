Claude Code徹底ガイド：Dynamic Workflow・モデル・工数（effort）の使い分け

Opus 4.8を普段使いする人向けに、「Dynamic Workflowで何ができるか」「モデル(Opus/Sonnet/Haiku/Fable)の使い分け」「工数(low〜max＋Ultracode)の選び方」を、2026年6月時点のAnthropic一次情報をもとに整理。

難易度: 応用 カテゴリ: AI・データ活用 更新日: 2026-06-17
Claude Code Dynamic Workflow サブエージェント 工数(effort) Ultracode モデル選択 Opus 4.8 Fable 5 xhigh レート制限

TL;DR

  • Dynamic Workflow = Claudeがその場でJavaScriptの台本を書き、裏で多数のサブエージェント（同時最大16・1実行で合計1,000まで）を束ねる仕組み。中間結果は台本に溜まり、会話の文脈には最終回答だけ返る。
  • 工数(effort)は「賢さ ↔ 速さ/節約」のスライダー。low / medium / high / xhigh / max ＋ Claude Code専用の ultracode。Opus 4.8の既定はhigh。
  • Ultracodeはモデルの工数レベルではなくClaude Codeの設定。xhigh＋「実質的なタスクごとにClaudeが自分でワークフローを組む」モード。最も徹底的・最も低速・消費が最大で、セッション限定。
  • 公式の原則は「モデルを変えるより工数を回す方が良いレバー」。Sonnet=日常の万能、Opus=難所、Haiku=単純/大量・サブエージェント、Fable=一回で終わらない大仕事。
  • 普段使い(Opus 4.8)の推奨: highのまま → 重い調査・長文比較・複雑な計画はxhigh → maxはほぼ不要 → Ultracodeは大規模並列だけ（消費最大なのでレート残量に注意）。

図解

1. Dynamic Workflowとは / 何ができるか

Dynamic Workflowは、Claudeが書いたJavaScriptの台本を専用ランタイムが裏で実行し、サブエージェントを大規模に束ねる機能です。台本がループ・分岐・中間結果を保持するので、自分の会話文脈は最終回答だけで済みます。つまり、文脈を汚さず大量処理できます。

向いているのは、1会話では捌けない規模の作業です。コードベース全体のバグ掃討、500ファイル級の一括移行、出典を相互照合するリサーチ、複数視点からの計画づくりが代表例です。

オーケストレーションの型（6つ）

  1. ファンアウト→集約→統合。
  2. パイプライン。各アイテムが段を独立して流れ、全件揃える時だけバリアを置く。
  3. 敵対的検証。1発見につき複数の懐疑エージェントに反証させ、過半数生存で採用する。
  4. 多視点検証。正しさ/セキュリティ/性能/再現性を別担当に分ける。
  5. 審査パネル。N案→並列ジャッジ→勝ち案から統合する。
  6. 枯れるまでループ。新発見がK回連続ゼロまで回し、既出は重複排除する。

制限・コスト

  • 同時実行は最大16体です。CPUコアが少ないと減ります。
  • 1実行で合計1,000体までです。これは暴走防止のための上限です。
  • 実行中はユーザー入力不可です。エージェントの権限確認だけが停止点になります。
  • 台本自身はファイル/シェルに直接触れません。読み書き・コマンドはサブエージェントが担当します。
  • サブエージェントは常にacceptEditsモードで動きツール許可リストを継承し、ファイル編集は自動承認されます。
  • トークン消費は通常セッションより大幅に多く、プラン使用量・レート制限を普通に消費します。まず狭い範囲(1ディレクトリ/絞った問い)で試し、/workflowsで消費を見て、重要でない段は小さいモデルに回すのが現実的です。

入口/使い方

入口は、プロンプトに ultracode キーワードを書く、自然文で「ワークフローで」と頼む、または /effort ultracode を使う、の3つです。

標準同梱の /deep-research <問い> は、複数角度でWeb検索し、出典を相互照合し、各主張に投票し、生き残った主張だけで出典付きレポートを作ります。自作台本は /workflows で s キー保存し、/<名前> で再利用できます。

要件は、Claude Code v2.1.154以降、全有料プランです。Proは /config でONにします。API・Amazon Bedrock・Google Vertex AI・Microsoft Foundryで利用できます。無効化は /config、settings.json の "disableWorkflows": true、環境変数 CLAUDE_CODE_DISABLE_WORKFLOWS=1 で行います。

2. 工数(effort)の使い分け

工数は、「どれだけ考え・ツールを呼び・説明するか」の行動シグナルです。厳密なトークン上限ではありません。低いほど速く安くレート消費も遅く、高いほど深く考え、ツール呼び出しも増えます。

low

最も効率的・最速最安。リネーム/誤字修正/grep/分類、サブエージェント向き。

medium

バランス寄り・コスト削減。

high

賢さと効率の釣り合い（既定）。

xhigh

より深い推論・消費増。コーディング/エージェント作業の推奨スタート点、探索・検索系。

max

上限なしの最大思考。フロンティア級のみ、過剰思考・効果逓減に注意。

ultracode

モデルのレベルではなくClaude Code設定。詳しくはセクション3。

対応モデルと既定値

Fable 5 / Opus 4.8 / Opus 4.7 は low/medium/high/xhigh/max。Opus 4.6 / Sonnet 4.6 は low/medium/high/max（xhighなし）。Haiku 4.5 は max非対応です。

既定値は、Fable 5・Opus 4.8・Opus 4.6・Sonnet 4.6 は high、Opus 4.7 は xhigh。high指定は「指定なし」と同じ挙動です。

永続性も違います。low/medium/high/xhigh はセッションをまたいで保持されます。max と ultracode は当該セッションのみで、新規セッションでリセットされます。スケールはモデルごとに別物なので、同じ"high"でも量は違います。

設定方法

/effort は引数なしでスライダーを開き、/effort xhigh で直接指定、/effort auto で既定に戻せます。/modelの左右キー、起動時 --effort、環境変数 CLAUDE_CODE_EFFORT_LEVEL、settings.jsonの effortLevel（low〜xhighのみ）、skill・subagentのフロントマターでも設定できます。優先順位は 環境変数 > 設定値 > モデル既定です。

単発で深く考えさせたい時は、プロンプトに ultrathink と書きます。その1ターンだけで、セッション設定やAPIのeffortは変えません。

公式指針

公式指針は「モデルを変えるより工数を回す方が良いレバー」です。Opus 4.8/4.7のコーディング・エージェント作業は xhigh から始め、賢さが要る作業は high が最低ライン、コスト重視は medium、max は xhigh で伸びしろが実測できた時だけにします。高い工数はレート制限を速く消費し、低い工数は速く安くレート消費も遅いので、サブスクのレート節約の第一レバーは工数を下げることです。

3. Ultracodeの正体

Ultracodeは、モデルの工数レベルではなくClaude Codeの設定です。xhighをモデルに送りつつ、「実質的なタスクごとにClaudeが自分の判断でワークフローを組む」常設許可を与えます。

素のxhighは推論を深めるだけで、ワークフローは自分で頼んで初めて発動します。Ultracodeはキーワード不要でClaudeが勝手に判断し、1つの依頼が「理解→変更→検証」と複数ワークフローに連鎖することもあります。

最も徹底的・最も低速・消費が最大です。普通の1ファイル編集には使わず、大規模で並列化が効くタスク、たとえばコードベース規模の移行、横断監査、相互照合リサーチに限ります。

全セッションで入れるなら /effort ultracode、単発なら ultracode: キーワードを使います。戻すときは /effort high です。

Ultracodeはセッション限定で、新しいセッションを始めるとリセットされます。タブを閉じる/アプリ再起動はその一例です。手動で /effort を変えても戻せます。xhigh対応モデルでのみ /effort メニューに出ます。

4. モデルの使い分け（2026-06時点）

モデル モデルID 価格（$/100万トークンの 入力/出力） 文脈・最大出力 主な使いどころ メモ
Fable 5 claude-fable-5 $10/$50 文脈1M・最大出力128K 一般提供で最強。一回で終わらない大仕事・曖昧な根本原因調査・長時間自律。 /model fable か別名 best。既定ではない。常にadaptive thinking（OFF不可）。
Opus 4.8 claude-opus-4-8 $5/$25 文脈1M・最大出力128K Opus級フラッグシップ・複雑推論の推奨既定。大規模リファクタ・繊細なバグ・設計・長期自律エージェント。 難所に寄せる時の主力。
Sonnet 4.6 claude-sonnet-4-6 $3/$15 文脈1M・最大出力64K 速度と知性のバランス。日常コーディングの万能既定・量産。 普段の作業を広く任せやすい。
Haiku 4.5 claude-haiku-4-5 $1/$5 文脈200K・最大出力64K 最速・最安・準フロンティア。単純/大量・分類・抽出・サブエージェント。 max・adaptive thinking非対応。

既定モデルはプラン依存

Max/Team Premium/Enterprise従量/API → Opus 4.8。Pro/Team Standard/Enterpriseシート → Sonnet 4.6。AWS版 → Opus 4.7。Bedrock/Vertex/Foundry → Sonnet 4.5。

選び方

Sonnetを日常の既定にし、詰まったらOpus、単純/大量はHaiku、1回で終わらない曖昧な難問はFable、という整理が実務では使いやすいです。opusplan=計画モードはOpus・実行は自動でSonnetに切替。品質が落ちたらモデル/工数を上げる前に /clear で文脈整理します。文脈が埋まると性能が落ちます。

5. 普段使い(Opus 4.8)の運用方針

場面 工数 運用メモ
ノート編集・要約・軽い整形 high（既定のまま） 単純作業は medium でレート節約も可。
深いリサーチ・長文の比較分析・複雑な計画 xhigh 調査や探索の質を優先する。
フロンティア級の難問（稀） max 常用しない・テストしてから。
横断リサーチ／一括監査／大量ファイル処理 Ultracode 消費最大なので残量を見て狭く絞る。

運用の順番

  1. /clearで文脈を整える。
  2. 一番難しい部分に合わせてモデル選択、または opusplan を使う。
  3. 難所はxhigh、品質が保てるならmedium/lowに落として節約する。
  4. max/Ultracodeはフロンティア・大規模並列だけにし、まず狭く試す。

Ultracodeとmaxは消費が跳ねるので残量を見て発動し、終わったら /effort high に戻します。

用語集

Dynamic Workflow

ClaudeがJavaScriptの台本を作り、専用ランタイムでサブエージェントを束ねる仕組み。中間結果を台本側に保持し、最終回答だけを会話文脈へ戻します。

サブエージェント

Dynamic Workflowの中で、調査・編集・検証などを担当する裏側の作業者。重要でない段は小さいモデルに回すと消費を抑えやすくなります。

工数(effort)

「どれだけ考え・ツールを呼び・説明するか」を調整する行動シグナル。厳密なトークン上限ではありません。

xhigh

より深い推論・消費増の工数レベル。Opus 4.8/4.7のコーディング・エージェント作業では推奨スタート点です。

Ultracode

モデルの工数レベルではなくClaude Codeの設定。xhigh＋「実質的なタスクごとにClaudeが自分でワークフローを組む」モードです。

adaptive thinking

モデル側の思考量を適応的に扱う仕組み。Fable 5は常にadaptive thinking（OFF不可）で、Haiku 4.5はadaptive thinking非対応です。

opusplan

計画モードはOpus・実行は自動でSonnetに切替する使い方。難しい設計と日常的な実行を分けたい時に使います。

/deep-research

標準同梱のワークフロー。複数角度でWeb検索し、出典を相互照合し、各主張に投票して、生き残った主張だけで出典付きレポートを作ります。
