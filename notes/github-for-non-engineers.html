Non-engineer guide

非エンジニアのためのGitHub入門 — 公開・実行の仕組みを理解する

GitHubは難しそうに見えるが、非エンジニアが触る範囲は意外と少ない。「ファイルを保管・公開する場所」という核を押さえれば、サイト公開もコード実行も同じ仕組みの上で理解できる。

GitHubとは何か

まずメンタルモデルを整える。4つの概念を押さえるだけで、ほとんどの説明が通じるようになる。

GitHub

ファイル一式（プロジェクト）を、変更履歴つきでインターネット上に保管・共有できるサービス。中心にあるのは「リポジトリ」という単位。

Microsoft が運営しており、世界中の開発者が使っている最大級のコード共有サービスでもある。

リポジトリ（repo）

1つのプロジェクト用のフォルダ。中身のファイルに加えて「いつ・誰が・何を変えたか」の履歴を丸ごと持つ。

ローカル（自分のPC）とGitHub上（クラウド）の両方に存在し、同期する仕組みになっている。

コミット（commit）

「ここまでの状態を1つのセーブポイントとして記録する」操作。短い説明文（コミットメッセージ）が付き、後から履歴をたどれる。

ゲームのセーブデータに近いイメージ。いつでも過去の状態に戻れる。

Git と GitHub の違い

Git はPC上で履歴を管理する仕組み（道具）。GitHub はそのGitリポジトリをネット上に置いて共有・公開できるようにするサービス（場所）。

混同しやすいが「道具」と「置き場所」の関係。Git なしでは GitHub は成り立たないが、Git 自体は GitHub がなくても使える。

public / private とブランチ

リポジトリの公開設定と、「ブランチ」という用語を最低限だけ整理する。

public リポジトリ

誰でも中身を見られる（コピーもできる）。GitHub Pages でサイト公開する場合は基本 public にする必要がある。

「公開」とはサイトだけでなく、元のファイル（コード・HTML）もネットから見える状態になることを意味する。機密情報を載せないことが鉄則。

private リポジトリ

自分（と招待した人）だけが見られる。公開したくないものはこちら。

ただし private リポジトリでは GitHub Pages の無料公開機能が使えない制約がある（有料プランなら可能）。

ブランチ（branch）

本線（main）を壊さずに作業するための「並行コピー」。変更を試してから本線に取り込む、という流れで使う。

非エンジニアは普段 main 1本だけで十分。「ブランチ」という言葉が出てきたとき用語として知っておく程度でよい。

main ブランチ

リポジトリの「本線」にあたる標準のブランチ名。以前は master と呼ばれていたが、現在は main が標準。

GitHub Pages の公開設定では「どのブランチを公開するか」を指定する。多くの場合 main を指定する。

ナレッジをWebサイトとして公開する仕組み（GitHub Pages）

リポジトリに置いたHTMLファイルがそのままWebサイトになる仕組みを、構造から理解する。

GitHub Pages とは

リポジトリに置いたHTML/CSS/JavaScriptのファイルを、GitHubが無料でWebサイトとしてインターネットにそのままホスティングしてくれる機能。

サーバーを自分で借りたり、契約したりする必要がない。リポジトリに push するだけで自動で公開される。

「静的サイト」とは

置いたファイルを「そのまま」配信する方式。アクセスのたびにサーバー側で計算したり、データベースを引いたりしない。

ナレッジ閲覧サイトのように「決まった内容を見せるだけ」の用途には十分で、無料・高速・壊れにくいという特長がある。

URLの構造と index.html

公開されたサイトのURLは https://<ユーザー名>.github.io/<リポジトリ名>/ という形になる。

URLにファイル名を付けずにフォルダを開くと、自動的に index.html が表示される。だから一覧ページは index.html という名前にする。

GitHub Pages の制約

静的ファイルのみ（サーバー側プログラムやデータベースは動かせない）。public リポジトリだと元のファイル自体もネットから見える。

大規模サイトや常時更新アプリには不向き。ナレッジ・ポートフォリオ・資料公開に最適な用途。

公開までのフロー

1. push する HTMLファイルをリポジトリに置く。ローカルの変更をGitHub上へ送る操作が「push」。
2. Pages を有効化 どのブランチ・どのフォルダを公開するか、リポジトリ設定で指定する。
3. 自動ビルド GitHubが公開用に自動で配置する。数十秒〜数分で反映される。
4. URL で公開 https://<ユーザー名>.github.io/<リポジトリ名>/ で世界中からアクセス可能になる。

このサイトで使っている設定

.nojekyll GitHub Pages の標準変換処理（Jekyll）をオフにして、置いたHTMLをそのまま配信させるための空ファイル。Jekyll を無効化しないと、アンダースコアで始まるフォルダが非表示になるなどの副作用がある。
robots.txt Disallow: / を書くことで、検索エンジンのロボットに「このサイトは巡回しないで」と伝えるファイル。URLを直接知っている人だけが見られる状態にするための設定。
noindex メタタグ 各HTMLの <head> に入れて「検索結果に載せないで」と指示するタグ。robots.txt と合わせることで、URLを知っている人だけが見られる"半非公開"状態にする。

公開コードを Claude Code / Codex で読み込み「動かす」原理

「やり方は知っているが原理は分かっていない」という部分を、3つの登場人物から整理する。

3つの登場人物

GitHub コードの「保管庫」。コードはここに置いてあるだけで、ここでは動かない。public リポジトリなら誰でもURLさえ分かればアクセスできる。
自分のPC（またはサンドボックス環境） コードが実際に「動く場所」。プログラムを実行するのはあくまで手元の環境。GitHub からファイルを持ってきて、ここで動かす。
Claude Code / Codex その2つを橋渡しして操作する「作業者」。人間の代わりにコマンドを打ち、ファイルを読み、実行する。魔法でネットのコードを覗くのではなく、一度手元に落としてからローカルのファイルとして読んでいる。

git clone の原理

clone とは

git clone <リポジトリのURL> は、「GitHub上のリポジトリを、ファイルも履歴も丸ごと自分のPCにコピーする」操作。

public リポジトリなら URL さえ分かれば誰でも clone できる。これが「ネット上のコードを手元に持ってくる」の正体。

ダウンロードとの違い

単純なファイルダウンロードに近いが、履歴も含めて完全な複製が手に入る点が特徴。

clone 後は GitHub との接続が残るため、後で git pull を実行すれば最新の変更を取り込むことができる。

「読み込ませる」とは — クローンからファイル読み取りまで

1. git clone 手元に GitHub のリポジトリをコピーする。ファイルも履歴も全部落ちてくる。
2. ローカルのファイルを開く Claude Code / Codex がローカルに落ちたファイルを開いて読む。ネット越しに読んでいるのではない。
3. 構造・処理を把握 コードの構造、関数の役割、依存関係などを解析して理解する。

「動作させる」とは — 実行環境の準備から起動まで

1. 言語と環境の確認 そのコードが何の言語か（Node.js / Python など）を確認し、対応する実行環境（ランタイム）が手元に入っているか確認する。
2. 依存ライブラリのインストール 他人が作った部品（ライブラリ）をインストールする。例: npm install、pip install。コードは単体では動かず、材料が揃って初めて動く。
3. 起動コマンドを実行 README 等の手順に従って起動コマンドを実行する。ここで初めてコードが「動く」状態になる。

なぜ「動くとき」と「動かないとき」があるのか

環境の違いが原因の大半

コード本体は同じでも、動かす側の環境（OS、言語のバージョン、入っているライブラリ）が違うと結果が変わる。

「自分のPCでは動かない」の多くは、コードの間違いではなく環境・依存のズレ。

Claude Code / Codex はズレを解消できる

このズレを読み取って、足りない物を入れたり、手順を調整したりしてくれるのが Claude Code / Codex の役割。

エラーメッセージを貼り付けると、何が足りないかを診断して対処方法を提案してくれる。

README / package.json / requirements.txt の役割

README

そのリポジトリの「取扱説明書」。何のコードで、どう動かすかが書いてある。Claude Code / Codex もまずここを読んで全体像を把握する。

package.json（Node.js）

「材料リスト」。このコードが必要とする外部部品（パッケージ）の一覧。npm install 時にこれを見て部品をそろえる。

requirements.txt（Python）

Python版の「材料リスト」。pip install -r requirements.txt で一括インストールできる。

セキュリティの注意 — 公開コードを動かすリスク

公開コードを動かす＝知らない人が書いたプログラムを自分の環境で実行すること。悪意あるコードのリスクがある。

  • 信頼できる出所かどうか確認してから動かす
  • 中身を Claude Code / Codex に確認させてから動かす
  • できれば隔離された環境（サンドボックス）で動かす

用語集

このノートで出てきた言葉を、最小限の文脈で整理する。

GitHub

Gitリポジトリをネット上に置いて共有・公開できるサービス。世界最大のコード共有プラットフォーム。

リポジトリ

1つのプロジェクト用のフォルダ単位。ファイルと変更履歴を丸ごと管理する入れ物。

コミット

変更内容をセーブポイントとして記録する操作。コミットメッセージ（説明）が付き、後から履歴をたどれる。

push

ローカル（自分のPC）でのコミットをGitHub上に送って反映させる操作。GitHub Pages の更新もこれで行う。

clone

GitHub上のリポジトリをファイルも履歴も丸ごと自分のPCにコピーする操作。git clone <URL> で実行。

branch（ブランチ）

本線（main）を壊さずに作業するための並行コピー。非エンジニアは main 1本だけで通常は十分。

main

リポジトリの本線にあたる標準ブランチ名。GitHub Pages の公開設定で指定することが多い。

GitHub Pages

リポジトリに置いたHTMLファイルを無料でWebサイトとして公開できる GitHub の機能。サーバー契約不要。

静的サイト

置いたファイルをそのまま配信する方式のWebサイト。サーバー側での計算やDB参照がない。高速・安定・無料向き。

ランタイム

プログラムを実際に動かすために必要な実行環境。Node.js や Python がこれにあたる。ランタイムがないとコードは動かない。

依存ライブラリ

コードが動くために必要な外部部品のこと。package.json や requirements.txt に一覧が書かれており、インストールして揃える。

README

リポジトリの取扱説明書。何のコードで、どう動かすかが書かれている。Claude Code / Codex も最初にここを読む。

public / private

リポジトリの公開設定。public は誰でも見られる（元のファイルも含む）、private は招待した人だけが見られる。
